1. Completar código


class Contador extends StatefulWidget {
  @override
  _ContadorState createState() => _ContadorState();
}

class _ContadorState extends State<Contador> {
  int contador = 0;

  void incrementar() {
    setState(() {
      contador++;
    });
  }

  @override
  Widget build(BuildContext context) {
    return Column(
      children: [
        Text('Valor: $contador'),
        ElevatedButton(
          onPressed: incrementar,
          child: Text('Sumar'),
        ),
      ],
    );
  }
}

Explicación: Se usa StatefulWidget porque el estado cambia. setState fuerza el rebuild de la UI.

2. Detección de errores


class Tareas extends StatefulWidget {
  @override
  _TareasState createState() => _TareasState();
}

class _TareasState extends State<Tareas> {

  List<String> tareas = [];

  void addTarea() {
    setState(() {
      tareas.add("Nueva tarea");
    });
  }

  @override
  Widget build(BuildContext context) {
    return Column(
      children: [
        ElevatedButton(
          onPressed: addTarea,
          child: Text("Añadir"),
        ),
        Expanded(
          child: ListView.builder(
            itemCount: tareas.length,
            itemBuilder: (context, index) {
              return Text(tareas[index]);
            },
          ),
        )
      ],
    );
  }
}

Errores originales:

  • Uso incorrecto de StatelessWidget con estado mutable
  • Falta de setState para actualizar UI
  • ListView dentro de Column sin Expanded (overflow)

Corrección aplicada: StatefulWidget + setState + Expanded.

3. Estructura de datos


class Tarea {
  String texto;
  bool completada;

  Tarea(this.texto, this.completada);
}

List<Tarea> tareas = [
  Tarea("Comprar", false),
  Tarea("Estudiar", true),
];

Cambio de estado:


setState(() {
  tareas[index].completada = true;
});

Justificación: Se usa objeto porque permite escalar atributos sin depender de Strings.

class Detalle extends StatelessWidget {
  final Tarea tarea;

  Detalle({required this.tarea});

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      body: Column(
        children: [
          Text(tarea.texto),
          Text(tarea.completada ? "Completada" : "Pendiente"),
        ],
      ),
    );
  }
}

5. Razonamiento

Si eliminamos setState(), Flutter no reconstruye el árbol de widgets, por lo tanto la interfaz no refleja los cambios del estado aunque los datos internos sí cambien.

Conclusión: setState es esencial para sincronizar estado y UI.
