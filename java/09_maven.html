    Maven, Dependencias y Gestión de Proyectos en Java

    1. ¿Qué es Maven?

    Maven es una herramienta de automatización y gestión de proyectos Java. Permite compilar, probar, empaquetar y distribuir proyectos de forma consistente y automatizada.

    1.1. Funciones principales de Maven

    • Gestión de dependencias: Descarga e integra librerías automáticamente
    • Compilación: Compila el código Java de forma automática
    • Testing: Ejecuta tests unitarios automáticamente
    • Empaquetado: Crea archivos JAR, WAR o EAR
    • Construcción: Automatiza todo el ciclo de vida del proyecto
    • Documentación: Genera sitios de documentación automáticamente

    1.2. Ventajas de usar Maven

    • Estructura estándar de proyectos (convención sobre configuración)
    • Gestión centralizada de dependencias
    • Ciclo de vida del proyecto bien definido
    • Integración con IDEs (Eclipse, IntelliJ, NetBeans)
    • Disponibilidad de miles de dependencias en Maven Central
    • Fácil distribución y compartición de proyectos

    1.3. Maven vs otras herramientas

    • Ant: Más antiguo, requiere más configuración manual
    • Gradle: Más moderno, más flexible, sintaxis con Groovy
    • Maven: Estándar de la industria, muy documentado

    2. Instalación de Maven

    2.1. Requisitos previos

    • Java JDK 1.7 o superior instalado
    • Variable de entorno JAVA_HOME configurada
    • Acceso a internet (para descargar dependencias)

    2.2. Instalación en Windows

    Sigue estos pasos:

    1. Descarga Maven desde https://maven.apache.org/download.cgi
    2. Descomprime el archivo en una carpeta (ej: C:\maven)
    3. Agrega C:\maven\bin a la variable PATH del sistema
    4. Verifica: Abre CMD y ejecuta mvn --version

    2.3. Instalación en macOS/Linux

    
# Opción 1: Descargar manualmente
wget https://archive.apache.org/dist/maven/maven-3/3.8.6/binaries/apache-maven-3.8.6-bin.tar.gz
tar xzf apache-maven-3.8.6-bin.tar.gz
sudo mv apache-maven-3.8.6 /opt/maven

# Opción 2: Usando package manager (macOS)
brew install maven

# Opción 3: Usando package manager (Linux - Ubuntu/Debian)
sudo apt-get install maven

# Verifica la instalación
mvn --version

    2.4. Configurar JAVA_HOME

    
# En Linux/macOS - Añade al archivo ~/.bashrc o ~/.zshrc
export JAVA_HOME=/usr/libexec/java_home
export PATH=$JAVA_HOME/bin:$PATH

# En Windows - Crea variable de entorno
# JAVA_HOME = C:\Program Files\Java\jdk1.8.0_201

    2.5. Verificar la instalación

    
mvn --version
# Apache Maven 3.8.6
# Maven home: /opt/maven
# Java version: 11.0.15

    3. Estructura de un proyecto Maven

    3.1. Estructura estándar

    
mi-proyecto/
├── src/
│   ├── main/
│   │   ├── java/              # Código fuente principal
│   │   │   └── com/ejemplo/
│   │   │       └── Main.java
│   │   ├── resources/         # Archivos de configuración
│   │   │   └── application.properties
│   │   └── webapp/            # Para proyectos web (WEB-INF, JSP, etc.)
│   └── test/
│       ├── java/              # Código de tests
│       │   └── com/ejemplo/
│       │       └── MainTest.java
│       └── resources/         # Recursos para tests
├── pom.xml                    # Archivo de configuración principal
├── target/                    # Archivos compilados (generado)
│   ├── classes/
│   ├── test-classes/
│   └── mi-proyecto-1.0.jar
└── README.md

    3.2. Convención sobre configuración

    Maven utiliza convención sobre configuración. Esto significa que asume una estructura estándar y no requiere configuración explícita para directorios:

    • Código fuente: src/main/java
    • Recursos: src/main/resources
    • Tests: src/test/java
    • Archivos compilados: target
    • JAR final: target/nombreproyecto.jar

    3.3. Crear un proyecto rápidamente

    
# Crear proyecto usando un archetype (plantilla)
mvn archetype:generate -DgroupId=com.ejemplo -DartifactId=mi-app -DarchetypeArtifactId=maven-archetype-quickstart -DinteractiveMode=false

# Esto crea la estructura completa automáticamente
cd mi-app

    4. Archivo pom.xml

    El archivo pom.xml (Project Object Model) es el corazón de cualquier proyecto Maven. Define las dependencias, plugins y configuración del proyecto.

    4.1. Estructura básica de pom.xml

    
<?xml version="1.0" encoding="UTF-8"?>
<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 
         http://maven.apache.org/xsd/maven-4.0.0.xsd">
    <modelVersion>4.0.0</modelVersion>

    <groupId>com.ejemplo</groupId>
    <artifactId>mi-proyecto</artifactId>
    <version>1.0.0</version>
    <packaging>jar</packaging>

    <name>Mi Proyecto</name>
    <description>Descripción del proyecto</description>

    <properties>
        <maven.compiler.source>11</maven.compiler.source>
        <maven.compiler.target>11</maven.compiler.target>
        <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
    </properties>

    <dependencies>
        <!-- Aquí van las dependencias -->
    </dependencies>

    <build>
        <plugins>
            <!-- Aquí van los plugins -->
        </plugins>
    </build>
</project>

    4.2. Explicación de elementos principales

    • groupId: Identificador único del grupo (similar a package Java)
    • artifactId: Nombre único del proyecto dentro del grupo
    • version: Versión del proyecto (ej: 1.0.0)
    • packaging: Tipo de empaquetado (jar, war, ear)
    • properties: Variables reutilizables en el pom.xml
    • dependencies: Lista de dependencias del proyecto
    • build: Configuración de compilación y plugins

    4.3. Identificador único (GAV)

    GroupId + ArtifactId + Version forman un identificador único: 

    
com.ejemplo:mi-proyecto:1.0.0
  ↑           ↑           ↑
groupId   artifactId   version

    5. Gestión de dependencias

    5.1. Añadir dependencias

    
<dependencies>
    <!-- JUnit para testing -->
    <dependency>
        <groupId>junit</groupId>
        <artifactId>junit</artifactId>
        <version>4.13.2</version>
        <scope>test</scope>
    </dependency>

    <!-- Log4j para logging -->
    <dependency>
        <groupId>org.apache.logging.log4j</groupId>
        <artifactId>log4j-core</artifactId>
        <version>2.20.0</version>
    </dependency>

    <!-- JDBC Driver para MySQL -->
    <dependency>
        <groupId>mysql</groupId>
        <artifactId>mysql-connector-java</artifactId>
        <version>8.0.33</version>
    </dependency>
</dependencies>

    5.2. Scopes (Alcances) de dependencias

    • compile (default): Disponible en compilación, testing y runtime
    • test: Solo disponible en tests (ej: JUnit, Mockito)
    • runtime: Disponible en runtime, no en compilación
    • provided: Proporcionado por el contenedor (ej: servlet-api en Tomcat)
    • system: Hacer referencia a un JAR en el sistema de archivos

    5.3. Especificar versiones

    
<dependencies>
    <!-- Versión exacta -->
    <dependency>
        <groupId>org.junit.jupiter</groupId>
        <artifactId>junit-jupiter</artifactId>
        <version>5.9.0</version>
    </dependency>

    <!-- Rango de versiones -->
    <dependency>
        <groupId>commons-lang</groupId>
        <artifactId>commons-lang</artifactId>
        <version>[2.6,)</version> <!-- >= 2.6 -->
    </dependency>

    <dependency>
        <groupId>org.springframework</groupId>
        <artifactId>spring-core</artifactId>
        <version>[4.0,5.0)</version> <!-- >= 4.0 y < 5.0 -->
    </dependency>
</dependencies>

    5.4. Propiedades para versiones

    
<properties>
    <junit.version>5.9.0</junit.version>
    <spring.version>6.0.0</spring.version>
</properties>

<dependencies>
    <dependency>
        <groupId>org.junit.jupiter</groupId>
        <artifactId>junit-jupiter</artifactId>
        <version>${junit.version}</version>
    </dependency>

    <dependency>
        <groupId>org.springframework</groupId>
        <artifactId>spring-core</artifactId>
        <version>${spring.version}</version>
    </dependency>
</dependencies>

    6. Ciclo de vida de Maven

    6.1. Fases principales

    Maven tiene un ciclo de vida predefinido con varias fases. Ejecutar una fase ejecuta también todas las fases anteriores:

    • validate: Valida que el proyecto sea correcto
    • compile: Compila el código fuente
    • test: Ejecuta los tests unitarios
    • package: Empaqueta el código compilado (JAR, WAR, etc.)
    • integration-test: Ejecuta tests de integración
    • verify: Verifica que el paquete es válido
    • install: Instala el paquete en el repositorio local
    • deploy: Copia el paquete al repositorio remoto

    6.2. Ejecutar fases

    
# Compilar código
mvn compile

# Ejecutar tests
mvn test

# Empaquetar (compila, testa y empaqueta)
mvn package

# Instalar en repositorio local
mvn install

# Desplegar en repositorio remoto
mvn deploy

# Limpiar archivos generados
mvn clean

# Combinaciones útiles
mvn clean compile        # Limpia y compila
mvn clean test          # Limpia y ejecuta tests
mvn clean package       # Limpia, compila, testa y empaqueta
mvn clean install       # Limpia, compila, testa, empaqueta e instala

    6.3. Visualizar ciclo de vida

    
# Ver todas las fases disponibles
mvn help:describe -Dcmd=clean
mvn help:describe -Dcmd=compile

    7. Plugins y configuración de build

    7.1. Compilador Java

    
<build>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-compiler-plugin</artifactId>
            <version>3.10.1</version>
            <configuration>
                <source>11</source>
                <target>11</target>
                <encoding>UTF-8</encoding>
            </configuration>
        </plugin>
    </plugins>
</build>

    7.2. Plugin Surefire (Testing)

    
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-surefire-plugin</artifactId>
    <version>3.0.0-M8</version>
    <configuration>
        <includes>
            <include>**/*Test.java</include>
            <include>**/*Tests.java</include>
        </includes>
    </configuration>
</plugin>

    7.3. Plugin Jar (Empaquetado)

    
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-jar-plugin</artifactId>
    <version>3.3.0</version>
    <configuration>
        <archive>
            <manifest>
                <mainClass>com.ejemplo.Main</mainClass>
            </manifest>
        </archive>
    </configuration>
</plugin>

    7.4. Plugin Assembly (Empaquetar con dependencias)

    
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-assembly-plugin</artifactId>
    <version>3.4.2</version>
    <configuration>
        <descriptorRefs>
            <descriptorRef>jar-with-dependencies</descriptorRef>
        </descriptorRefs>
        <archive>
            <manifest>
                <mainClass>com.ejemplo.Main</mainClass>
            </manifest>
        </archive>
    </configuration>
    <executions>
        <execution>
            <phase>package</phase>
            <goals>
                <goal>single</goal>
            </goals>
        </execution>
    </executions>
</plugin>

    8. Repositorios y dependencias transitivas

    8.1. Repositorio local y remoto

    Maven mantiene una caché local de dependencias descargadas:

    • Repositorio local: ~/.m2/repository (tu máquina)
    • Repositorio remoto: Maven Central, repositorios corporativos

    8.2. Configurar repositorios personalizados

    
<repositories>
    <repository>
        <id>central</id>
        <url>https://repo.maven.apache.org/maven2</url>
    </repository>

    <repository>
        <id>repositorio-corporativo</id>
        <url>https://repo.miempresa.com/maven</url>
    </repository>
</repositories>

    8.3. Dependencias transitivas

    Si tu proyecto depende de Spring, y Spring depende de Commons Logging, Maven descarga automáticamente Commons Logging para ti: 

    
Tu proyecto
    ↓ depende de
Spring
    ↓ depende de
Commons Logging ← Maven la descarga automáticamente

    8.4. Visualizar árbol de dependencias

    
# Ver árbol de dependencias
mvn dependency:tree

# Ejemplo de salida:
# [INFO] +- org.springframework:spring-core:jar:6.0.0:compile

# [INFO] |  \- org.springframework:spring-aop:jar:6.0.0:compile
# [INFO] +- junit:junit:jar:4.13.2:test

    9. Perfiles (Profiles) en Maven

    9.1. ¿Qué son los perfiles?

    Los perfiles permiten tener diferentes configuraciones para diferentes contextos (desarrollo, testing, producción):

    9.2. Configurar perfiles

    
<profiles>
    <profile>
        <id>desarrollo</id>
        <activation>
            <activeByDefault>true</activeByDefault>
        </activation>
        <properties>
            <maven.compiler.debug>true</maven.compiler.debug>
        </properties>
    </profile>

    <profile>
        <id>produccion</id>
        <properties>
            <maven.compiler.debug>false</maven.compiler.debug>
            <maven.compiler.optimize>true</maven.compiler.optimize>
        </properties>
    </profile>
</profiles>

    9.3. Activar perfiles

    
# Activar perfil específico
mvn clean package -P produccion

# Activar múltiples perfiles
mvn clean package -P desarrollo,testing

# Ver perfiles activos
mvn help:active-profiles

    10. Gestión de archivos y recursos

    10.1. Archivos de configuración

    Los archivos en src/main/resources se copian automáticamente al JAR: 

    
src/main/resources/
├── application.properties
├── log4j2.xml
├── config/
│   └── database.properties
└── static/
    └── logo.png

    10.2. Leer archivos de configuración en Java

    
import java.io.InputStream;
import java.util.Properties;

public class Config {
    public static Properties cargarPropiedades() throws Exception {
        Properties props = new Properties();
        InputStream input = Config.class.getClassLoader()
            .getResourceAsStream("application.properties");
        props.load(input);
        return props;
    }
}

    10.3. Filtering (Reemplazos en recursos)

    
<build>
    <resources>
        <resource>
            <directory>src/main/resources</directory>
            <filtering>true</filtering>
        </resource>
    </resources>
</build>

<properties>
    <app.name>Mi Aplicación</app.name>
    <app.version>1.0.0</app.version>
</properties>

    En application.properties:

    
app.name=${app.name}
app.version=${app.version}

    11. Mejores prácticas con Maven

    11.1. Estructura de directorios

    • Siempre seguir la estructura estándar de Maven
    • Usar paquetes en Java siguiendo el patrón com.empresa.proyecto
    • Mantener tests en src/test/java
    • Recursos de configuración en src/main/resources

    11.2. Versionado de pom.xml

    • Usar versionado semántico: MAJOR.MINOR.PATCH
    • Especificar versiones explícitamente (evitar LATEST, RELEASE)
    • Usar propiedades para versiones comunes
    • Incluir pom.xml en control de versiones (git)

    11.3. Gestión de dependencias

    • Mantener dependencias actualizadas pero probadas
    • Usar scope adecuado para cada dependencia
    • Revisar regularmente dependencias transitivas
    • Evitar dependencias no utilizadas

    11.4. Testing con Maven

    
<dependencies>
    <dependency>
        <groupId>org.junit.jupiter</groupId>
        <artifactId>junit-jupiter</artifactId>
        <version>5.9.0</version>
        <scope>test</scope>
    </dependency>

    <dependency>
        <groupId>org.mockito</groupId>
        <artifactId>mockito-core</artifactId>
        <version>5.0.0</version>
        <scope>test</scope>
    </dependency>
</dependencies>

    11.5. Control de versiones (git)

    • Incluir: pom.xml, src/, README.md
    • Excluir: target/, .classpath, .project
    
# .gitignore recomendado para Maven
target/
.classpath
.project
.settings/
*.class
*.jar
*.war
*.ear
.DS_Store

    13. Resumen

    13.1. Puntos clave

    • Maven es el estándar de automatización en Java
    • pom.xml define todas las configuraciones del proyecto
    • Dependencias se descargan automáticamente del repositorio
    • Ciclo de vida define las fases de compilación, testing y empaquetado
    • Convención sobre configuración simplifica la configuración

    13.2. Flujo típico de trabajo

    1. Generar proyecto: mvn archetype:generate
    2. Definir dependencias en pom.xml
    3. Escribir código en src/main/java
    4. Escribir tests en src/test/java
    5. Compilar y probar: mvn clean test
    6. Empaquetar: mvn clean package
    7. Instalar: mvn clean install

    13.3. Estructura recomendada

    
proyecto/
├── src/
│   ├── main/
│   │   ├── java/com/ejemplo/
│   │   │   └── Main.java
│   │   └── resources/
│   └── test/
│       ├── java/com/ejemplo/
│       │   └── MainTest.java
│       └── resources/
├── pom.xml
├── README.md
└── .gitignore

    13.4. Comandos esenciales

    • mvn clean - Limpiar
    • mvn compile - Compilar
    • mvn test - Probar
    • mvn package - Empaquetar
    • mvn install - Instalar
    • mvn dependency:tree - Ver dependencias
