    Data Architecture and Engineering: The Data Lifecycle

    Module 1: Fundamentals, Data Types and Governance

    This module introduces the foundational concepts of data within modern organizations. It explores how raw data evolves into valuable insights and decisions, and how data flows through different stages from its creation to long-term storage.

    Understanding this lifecycle is essential to designing robust, scalable, and efficient data systems that support business objectives.

    Why Do Data Matter?

    DIKW Model: From Raw Data to Business Value

    Data, on its own, has little meaning. Its value emerges through transformation, context, and interpretation. The DIKW model explains how data is progressively refined into actionable insights that drive business decisions.

    • Data: Raw, unprocessed facts without context (e.g., "39.5")
    • Information: Data placed in context (e.g., "The patient's temperature is 39.5°C")
    • Knowledge: Interpretation of information (e.g., "The patient has a high fever")
    • Action/Value: Decisions and actions based on knowledge (e.g., prescribing treatment)

    This progression illustrates how organizations extract value from data by structuring processes that convert raw inputs into informed decisions.

    The Data Lifecycle

    Six Core Phases

    The data lifecycle describes the journey of data from its origin to its final storage or deletion. Each phase represents a critical step in ensuring data quality, availability, and usefulness.

    • Generation: Data is created by systems, devices, or users
    • Ingestion: Data is collected and transported into storage systems
    • Storage: Data is stored in databases, data lakes, or warehouses
    • Processing: Data is cleaned, transformed, and structured
    • Analysis: Data is explored and used for reporting or modeling
    • Archiving: Data is stored long-term or retired according to policies

    These phases form a continuous loop, enabling organizations to derive ongoing value from their data assets.

    Data Types

    Structured, Semi-structured and Unstructured Data

    Different types of data require different storage systems, processing techniques, and tools. Identifying the type of data early is essential for designing an efficient data architecture.

    • Structured Data: Highly organized with a fixed schema, typically stored in relational databases (e.g., tables with rows and columns)
    • Semi-structured Data: Does not follow a rigid schema but contains tags or markers (e.g., JSON, XML)
    • Unstructured Data: Lacks a predefined format (e.g., images, audio, videos, text documents)

    Choosing how to handle each type of data directly affects performance, scalability, and system complexity.

    Data Velocity

    Batch vs Real-Time Processing

    Data processing strategies depend on how quickly results are needed. Two main approaches are used depending on business requirements.

    • Batch Processing: Data is accumulated over time and processed periodically (e.g., daily reports)
    • Streaming Processing: Data is processed continuously as it is generated (e.g., real-time event handling)

    Batch processing is simpler and more cost-effective, while streaming enables immediate insights but increases system complexity and cost.

    Metadata: The DNA of Data

    Catalogs and Data Lineage

    Metadata provides essential context about data, enabling understanding, governance, and trust. Without metadata, managing data at scale becomes inefficient and error-prone.

    • Metadata: Describes data characteristics such as source, format, and meaning
    • Data Catalog: Centralized system for discovering and managing data assets
    • Data Lineage: Tracks the flow and transformations of data across systems

    Data lineage ensures transparency by showing how data moves from source systems to final reports, supporting auditing and validation processes.
