Objetivo de la sesión

Repasar los conceptos importantes del módulo para llegar al examen con una visión clara, conectada y práctica.

  • No vamos a “adivinar preguntas”.
  • Sí vamos a identificar ideas clave, relaciones entre unidades y errores típicos.
  • La prioridad es saber razonar las respuestas, no memorizar frases aisladas.

Mapa general del examen

El temario trabaja una progresión lógica:

  1. Sistema informático y software.
  2. Entornos de desarrollo.
  3. Pruebas y depuración.
  4. Optimización, mantenimiento y documentación.
  5. Modelado estructural con UML.
  6. Modelado dinámico del comportamiento.

¿Cómo será el examen?

El examen será de tipo test, con una única respuesta correcta por pregunta.

  • Total: 30 preguntas.
  • Opciones: 4 respuestas posibles: A, B, C y D.
  • Puntuación máxima: 10 puntos.
  • Respuesta correcta: suma 0,33 puntos.
  • Respuesta incorrecta: cada 3 errores restan el equivalente a 1 respuesta correcta.
  • Respuesta en blanco: no suma ni resta.
  • Forma de responder: marcar con una X en la hoja de respuestas.

Consejo: primero descarta opciones técnicamente imposibles antes de arriesgar.

Estrategia para un test técnico

  • Lee el enunciado completo: suele dar el contexto.
  • Descarta opciones imposibles o contradictorias.
  • Busca palabras clave: compilación, depuración, herencia, actor, estado, commit.
  • Si dudas entre dos, piensa qué problema resuelve cada concepto.

UT1 · Fundamentos del desarrollo de software

Conceptos clave:

Sistema informático, tipos de software, ciclo de vida, código fuente/objeto/ejecutable, máquinas virtuales y lenguajes.

1. Sistema informático: hardware + software

  • CPU: interpreta y ejecuta instrucciones.
  • Memoria: almacena datos e instrucciones; RAM para ejecución y almacenamiento para persistencia.
  • Periféricos: entrada, salida y almacenamiento.
  • Sistema operativo: coordina recursos y hace de intermediario entre aplicaciones y hardware.

Pregunta guía: cuando abro un archivo, ¿quién coordina la petición, la lectura del disco y la aplicación que lo muestra?

Relación con el examen real: P1.

2. Clasificación funcional del software

  • Sistema: sistema operativo, controladores.
  • Aplicación: programas orientados a tareas del usuario.
  • Programación: lenguajes, compiladores, IDEs.
  • Utilidades: antivirus, copias de seguridad, optimizadores.

Error típico: confundir una aplicación de usuario con software de sistema.

Relación con el examen real: P4.

3. Del código fuente al ejecutable

  • Código fuente: escrito por el programador; legible por humanos.
  • Código objeto: resultado intermedio de la compilación; aún no es el producto final.
  • Ejecutable: archivo final que puede ejecutar el sistema operativo tras compilación/enlazado.

Secuencia mental: escribir → compilar → enlazar/generar → ejecutar.

Relación con el examen real: P2.

4. Máquinas virtuales y código intermedio

Algunos lenguajes generan un código intermedio ejecutado sobre una máquina virtual.

  • Java: fuente → bytecode → JVM.
  • .NET: fuente → IL → CLR.
  • Python: fuente → bytecode → PVM/intérprete.

Idea clave: portabilidad entre plataformas si existe la máquina virtual adecuada.

Relación con el examen real: P3.

5. Clasificación de lenguajes

  • Por nivel: bajo, medio y alto nivel.
  • Por paradigma: imperativo, orientado a objetos, funcional, lógico.
  • Por ejecución: compilados, interpretados o con código intermedio.

Ejemplo: C suele asociarse a compilación; Python a interpretación; Java a POO y bytecode.

Relación con el examen real: P5.

UT2 · Instalación y configuración de entornos

IDE, configuración, plugins, automatización e integración con Git.

1. IDE: concepto y ejemplos

  • Integra editor, compilador/intérprete, depurador, gestión de proyectos y control de versiones.
  • Eclipse: libre y extensible.
  • Visual Studio: propietario, con versión Community gratuita.
  • PyCharm Community: gratuito para Python.
  • IntelliJ IDEA: Community y Ultimate.
Relación con el examen real: P6 y P10.

2. Productividad en el IDE

  • Personalizar temas, fuentes y paneles mejora el flujo de trabajo sin cambiar el código.
  • Los atajos reducen tiempo y errores repetitivos.
  • Los plugins añaden capacidades: Git, bases de datos, frameworks, Docker, análisis de código.
Relación con el examen real: P8.

3. Automatización y Git

  • Los scripts automatizan tareas repetitivas: crear proyectos, instalar dependencias, compilar.
  • Git permite controlar versiones, colaborar y recuperar cambios.
  • Operaciones esenciales: clone, status, add, commit, push, pull, branch, merge.

Clave: commit guarda una instantánea local; push la envía al remoto; pull trae cambios.

Relación con el examen real: P7, P9 y P19.

UT3 · Pruebas y depuración

Las pruebas detectan fallos; la depuración ayuda a analizarlos y corregirlos.

1. Tipos de pruebas

  • Unitarias: funciones o métodos aislados.
  • Funcionales: funcionalidades desde el punto de vista del usuario/requisitos.
  • Integración: varios módulos funcionando juntos.
  • Regresión: cambios nuevos no rompen lo anterior.
Relación con el examen real: P11, P12 y P15.

2. Depuración en el IDE

  • Breakpoint: detiene la ejecución en una línea concreta.
  • Inspección de variables: permite ver valores y estado de objetos.
  • Step over: avanza sin entrar en métodos.
  • Step into: entra en el método llamado.
  • Step return: sale del método actual.
Relación con el examen real: P13.

3. Automatización de pruebas

JUnit y PyTest permiten ejecutar verificaciones de forma sistemática y repetible.

@Test
public void testSumar() {
    int resultado = sumar(2, 3);
    assertEquals(5, resultado);
}

Idea clave: una prueba automatizada comprueba un resultado esperado sin intervención manual.

Relación con el examen real: P15.

4. Documentación de incidencias

  • Descripción clara del error.
  • Pasos para reproducirlo.
  • Entorno de prueba.
  • Resultado esperado y obtenido.
  • Gravedad, prioridad y estado.
Relación con el examen real: P14.

UT4 · Optimización y mantenimiento del código

Mejorar código no siempre significa añadir funciones: muchas veces significa hacerlo más claro, simple y mantenible.

1. Refactorización

Modificar la estructura interna del código sin cambiar su comportamiento externo.

  • Renombrar variables y métodos.
  • Extraer métodos.
  • Eliminar código duplicado.
  • Simplificar condicionales.
  • Modularizar con clases y métodos claros.
Relación con el examen real: P16.

2. Principios de mejora

  • DRY: no repetir lógica; extraer lo común.
  • KISS: mantener soluciones simples y comprensibles.
  • YAGNI: no implementar funcionalidad que aún no se necesita.
Relación con el examen real: P17.

3. Análisis estático y documentación

  • SonarQube: análisis de calidad en múltiples lenguajes.
  • ESLint: análisis centrado en JavaScript y derivados.
  • Javadoc: documentación para Java.
  • Sphinx: documentación habitual en Python.
Relación con el examen real: P18 y P20.

UT5 · Modelado orientado a objetos

UML ayuda a representar la estructura del sistema antes o durante la implementación.

1. Clase, objeto y POO

  • Clase: plantilla que define atributos y métodos.
  • Objeto: instancia concreta de una clase, con valores propios.
  • Herencia: una clase reutiliza y especializa otra.
  • Polimorfismo: una misma operación puede comportarse de formas distintas.
Relación con el examen real: P21.

2. Visibilidad UML

  • + público: accesible desde cualquier clase.
  • - privado: accesible solo desde la propia clase.
  • # protegido: accesible desde la clase y sus subclases.
  • ~ paquete: accesible dentro del mismo paquete.
Relación con el examen real: P22 y P25.

3. Relaciones en diagramas de clases

  • Asociación: relación general entre clases.
  • Agregación: “tiene un”, parte independiente; rombo blanco.
  • Composición: parte con ciclo de vida dependiente; rombo negro.
  • Herencia: generalización/especialización; flecha con triángulo blanco.
  • Dependencia: uso puntual; línea discontinua.
Relación con el examen real: P23 y P30.

4. Ingeniería directa e inversa

  • Ingeniería directa: generar código a partir de un modelo UML.
  • Ingeniería inversa: generar diagramas a partir de código existente.

Importante: la herramienta genera estructura; la lógica de negocio debe desarrollarse y revisarse.

Relación con el examen real: P24.

UT6 · Modelado dinámico

Los diagramas de comportamiento muestran qué hace el sistema y cómo evoluciona o interactúa en el tiempo.

1. Casos de uso

  • Representan funcionalidades desde la perspectiva de actores externos.
  • Elementos: actor, caso de uso, asociación, include, extend, generalización.
  • Responden a: ¿qué ofrece el sistema al usuario?
Relación con el examen real: P26.

2. Interacción: secuencia y comunicación

  • Secuencia: enfatiza el orden temporal de los mensajes.
  • Comunicación: enfatiza relaciones entre participantes y mensajes.

Pregunta guía: si lo importante es “qué mensaje ocurre primero y cuál después”, piensa en secuencia.

Relación con el examen real: P27.

3. Actividades y estados

  • Actividad: flujo de trabajo, acciones, decisiones, rutas alternativas y paralelismo.
  • Estados: ciclo de vida de un objeto, eventos y transiciones.

Ejemplo mental: un pedido puede pasar por creado, pagado, enviado, entregado o cancelado.

Relación con el examen real: P28.

4. Herramientas UML

  • Visual Paradigm, StarUML, Enterprise Architect, Modelio, ArgoUML.
  • Permiten modelar diagramas, documentar, exportar y en algunos casos generar o sincronizar código.

Clave: una herramienta ayuda, pero no sustituye a un modelo correcto.

Relación con el examen real: P29.

Errores típicos que conviene evitar

  • Confundir prueba con depuración.
  • Confundir código objeto con ejecutable final.
  • Confundir agregación y composición.
  • Creer que refactorizar es añadir funcionalidad.
  • Usar “diagrama de clases” para representar procesos o flujos.
  • Olvidar que Git commit es local y push es remoto.
Relación con el examen real: repaso transversal de P1 a P30.

Cierre: cómo estudiar las últimas horas

  • Repasa definiciones con ejemplos, no solo de memoria.
  • Haz comparativas: compilado vs interpretado, prueba vs depuración, clase vs objeto.
  • Dibuja mentalmente los símbolos UML básicos.
  • Practica descartar opciones falsas por contradicción técnica.
  • Si no sabes una respuesta, no improvises: razona desde el concepto.
