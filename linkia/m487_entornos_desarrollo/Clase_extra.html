UT1 · Preguntas 1–5

Fundamentos del Desarrollo de Software

5 preguntas del examen provienen de esta unidad

🎯

Los temas más preguntados: rol del SO como intermediario, diferencia código objeto/ejecutable, máquinas virtuales para portabilidad, clasificación del software y lenguajes compilados + paradigma OO.

Pregunta 1 · El sistema operativo como intermediario ⭐ examen

Cuando haces doble clic en un archivo, el SO:

  • Gestiona la solicitud del usuario
  • Recupera el archivo del almacenamiento (disco)
  • Lo envía a la aplicación adecuada (p.ej. un visor de imágenes)
Clave: El SO no compila ni transforma la CPU; actúa como gestor de recursos entre hardware y software.

Pregunta 2 · Código fuente → objeto → ejecutable ⭐ examen

TipoQué esSe puede ejecutar directamente
Código fuenteEl que escribe el programador (legible)No
Código objetoResultado de la compilaciónNo (falta enlazado)
Código ejecutableResultado del enlazadoSí ✓
Clave: código objeto = compilación · ejecutable = compilación + enlazado.

Pregunta 3 · Máquinas virtuales y portabilidad ⭐ examen

Para ejecutar una app en distintas plataformas sin recompilar para cada una:

Solución: Generar código intermedio (bytecode) que luego ejecuta una máquina virtual (JVM en Java, CLR en .NET). Esto abstrae el SO específico.

SoftwareTipoEjemplos

App de facturaciónAplicaciónSage, SAP

Pregunta 5 · Lenguajes compilados + paradigma OO ⭐ examen

  • Lenguajes compilados: traducen todo el código antes de ejecutarlo (≠ interpretados, que lo hacen línea a línea).
  • Paradigma orientado a objetos: modela el software mediante objetos con atributos y métodos.
Recuerda: Los compilados NO se traducen línea a línea (eso es el intérprete). El paradigma OO NO evita las instrucciones secuenciales.

Sin modificar el código fuente se puede mejorar el entorno:

UT3 · Preguntas 11–15

Pruebas y depuración en el desarrollo de aplicaciones

5 preguntas del examen

Pregunta 11 · Pruebas vs depuración ⭐ examen

ActividadObjetivo
PruebasDetectar fallos de forma sistemática
DepuraciónAnalizar y corregir los errores encontrados
Son complementarias, no excluyentes. Las pruebas detectan → la depuración corrige. La depuración NO sustituye a las pruebas.

Pregunta 12 · Prueba unitaria ⭐ examen

Comprobar una función sumar(a, b) de forma aislada mediante un framework = prueba unitaria.

Unitaria = unidad individual (función/método) · Funcional = funcionalidad completa · Aceptación = validación del cliente.

Pregunta 13 · Breakpoint (punto de ruptura) ⭐ examen

Para detener la ejecución en una línea concreta e inspeccionar variables y objetos → punto de ruptura (breakpoint). No es un commit, merge ni hook de Git.

Pregunta 14 · Documentar un fallo correctamente ⭐ examen

El conjunto mínimo para registrar un fallo y poder reproducirlo y priorizarlo:

  • Descripción del error
  • Pasos para reproducirlo
  • Entorno de prueba
  • Gravedad
  • Estado
No basta con el nombre del archivo ni con la solución propuesta ni con el historial de commits.

Pregunta 15 · JUnit y PyTest ⭐ examen

Ventaja compartida: Permiten automatizar pruebas y ejecutar verificaciones de forma sistemática. No sustituyen al IDE, no son para pruebas manuales ni están limitados a JavaScript.

JUnit → Java · PyTest → Python

UT4 · Preguntas 16–20

Optimización y Mantenimiento del Código

5 preguntas del examen

Pregunta 16 · Refactorización ⭐ examen

Definición exacta: Mejorar la estructura interna del código sin alterar su comportamiento externo. No añade funcionalidades nuevas. No convierte lenguajes.

Pregunta 17 · Principio DRY ⭐ examen

DRY = Don't Repeat Yourself. Si dos métodos hacen exactamente lo mismo:

Solución: Extraer la lógica común a un método compartido. Duplicar más código es exactamente lo contrario de DRY.

Pregunta 18 · SonarQube vs ESLint ⭐ examen

HerramientaLenguajesFunción principal
SonarQubeMúltiples lenguajesAnálisis de calidad de código, detecta bugs, vulnerabilidades y code smells
ESLintJavaScript y derivados (TS, JSX…)Linter: detecta errores de estilo y sintaxis en JS
Clave: SonarQube = múltiples lenguajes · ESLint = JavaScript/derivados. SonarQube SÍ puede integrarse con IDEs.

Modelado Orientado a Objetos: Diagramas de Clases

5 preguntas del examen

Pregunta 21 · Clase vs Objeto ⭐ examen

Clase = plantilla/molde que define estructura y comportamiento.
Objeto = instancia concreta con valores propios de una clase.
Ejemplo: Coche es la clase · miToyota es el objeto.

- marca: String - modelo: String

Preguntas 22 y 25 · Visibilidades en UML ⭐ examen

+
Público
Acceso desde cualquier clase del proyecto
−
Privado
Solo desde la propia clase
#
Protegido
Desde la clase y sus subclases (herencia)
~
Paquete
Desde clases del mismo paquete
P22: acceso desde propia clase y subclases → # protegido · P25: acceso desde cualquier clase → + público

Pregunta 23 · Relaciones entre clases ⭐ examen

Asociación
———
Relación genérica. Las clases se conocen pero no hay dependencia de ciclo de vida.
Composición ⭐
◆——— (rombo negro)
Propiedad fuerte. Las partes no existen sin el todo. Comparten ciclo de vida. Ej: Coche compone Motor.
Agregación
◇——— (rombo blanco)
Relación "tiene un" débil. Las partes pueden existir sin el todo.
Dependencia
- - - -→
Uso temporal de una clase por otra. Línea discontinua con flecha.
Coche compone Motor + ciclo de vida compartido = Composición (rombo negro).

Pregunta 24 · Ingeniería directa ⭐ examen

Ingeniería directa (forward engineering): obtener código fuente a partir de un diagrama UML.
Ingeniería inversa: obtener el diagrama a partir del código existente.

Pregunta 30 · Flecha con punta blanca (triángulo hueco) ⭐ examen

Flecha con punta blanca (triángulo hueco) = Herencia / Generalización. Va desde la subclase hacia la superclase.

Pregunta 29 · Software para diagramas UML ⭐ examen

StarUML es una herramienta real para diagramas UML (multiplataforma, open source). MagicUML, UMLPlus y UMLGenerator son inventados.
UT6 · Preguntas 26–30

Modelado Dinámico: Diagramas de Comportamiento

5 preguntas del examen

Pregunta 26 · Diagramas de casos de uso ⭐ examen

Representan las funcionalidades del sistema desde la perspectiva del actor externo (usuario). Describen qué hace el sistema, no cómo.

No representan código objeto, ni historial de commits, ni estructura de clases.

Pregunta 27 · Secuencia vs Comunicación ⭐ examen

DiagramaÉnfasisEjes / Estructura
SecuenciaOrden temporal de los mensajesEje vertical = tiempo · Eje horizontal = participantes
ComunicaciónRelaciones estructurales entre participantesSin eje temporal, muestra los enlaces entre objetos
Clave: Secuencia = tiempo · Comunicación = estructura de relaciones. El diagrama de comunicación no muestra solo estados (eso son los diagramas de estados).

Pregunta 28 · Flujo con decisiones binarias y rutas alternativas ⭐ examen

Para modelar acciones, decisiones binarias y rutas alternativas → Diagrama de actividades.
No es un diagrama de clases (estructura), ni de componentes, ni de despliegue.

Resumen tipos de diagramas UML

DiagramaPara qué sirve
ClasesEstructura estática: clases, atributos, relaciones
Casos de usoFuncionalidades desde perspectiva del actor
SecuenciaOrden temporal de mensajes entre objetos
ComunicaciónRelaciones estructurales en una interacción
ActividadesFlujos con acciones, decisiones y rutas alternativas ⭐
EstadosCiclo de vida de un objeto (estados y transiciones)
ComponentesEstructura de componentes software y sus dependencias
DespliegueDistribución física del sistema en nodos hardware
