    Composer, Autoload y PSR-4 en PHP

    1. ¿Qué es Composer?

    Composer es un gestor de dependencias para PHP. Permite instalar, gestionar y actualizar las librerías y paquetes que tu proyecto necesita de forma automática y centralizada.

    1.1. ¿Por qué usar Composer?

    • Gestión centralizada: Todas las dependencias en un archivo composer.json
    • Instalación automática: Descarga todas las librerías necesarias automáticamente
    • Versionado: Control de versiones de dependencias
    • Autoloading: Carga automática de clases mediante PSR-4
    • Actualizaciones: Gestiona actualizaciones de dependencias fácilmente
    • Evita duplicados: Cada dependencia se descarga una sola vez

    1.2. Composer vs otros métodos

    Antes de Composer, los desarrolladores PHP:

    • Descargaban archivos .zip manualmente
    • Usaban require() o include() para cargar archivos individuales
    • Tenían que resolver manualmente las dependencias entre librerías
    • No tenían control de versiones consistente

    Composer resuelve todos estos problemas automáticamente.

    2. Instalación de Composer

    2.1. Requisitos previos

    • PHP 5.3.2 o superior instalado
    • Acceso a internet (para descargar paquetes)
    • Línea de comandos (terminal/CMD)

    2.2. Instalación en Windows

    La forma más fácil es descargar el instalador desde https://getcomposer.org

    • Descarga el archivo Composer-Setup.exe
    • Ejecuta el instalador
    • Sigue los pasos del asistente de instalación
    • Verifica la instalación abriendo CMD y ejecutando: composer --version

    2.3. Instalación en macOS/Linux

    Abre la terminal y ejecuta los siguientes comandos:

    
# Descargar el instalador
php -r "copy('https://getcomposer.org/installer', 'composer-setup.php');"

# Ejecutar el instalador
php composer-setup.php

# Mover Composer a una ubicación global
sudo mv composer.phar /usr/local/bin/composer

# Verificar la instalación
composer --version

    2.4. Verificar la instalación

    
composer --version
# Resultado: Composer version 2.x.x 2024-xx-xx

    3. Archivo composer.json

    El archivo composer.json es el corazón de cualquier proyecto que usa Composer. Define las dependencias y configuración de tu proyecto.

    3.1. Crear un nuevo proyecto

    
# Crea un directorio para el proyecto
mkdir mi-proyecto
cd mi-proyecto

# Inicializar Composer
composer init

# Se te pedirá información:
# - Nombre del proyecto
# - Descripción
# - Autor
# - Licencia
# - Dependencias (initially empty)

    3.2. Estructura básica de composer.json

    
{
    "name": "nombre/proyecto",
    "description": "Descripción del proyecto",
    "type": "project",
    "require": {
        "php": ">=7.4",
        "monolog/monolog": "^2.0",
        "symfony/console": "^5.0"
    },
    "require-dev": {
        "phpunit/phpunit": "^9.0",
        "squizlabs/php_codesniffer": "^3.0"
    },
    "autoload": {
        "psr-4": {
            "App\\": "src/"
        }
    },
    "license": "MIT",
    "authors": [
        {
            "name": "Tu Nombre",
            "email": "tu@email.com"
        }
    ]
}

    3.3. Partes principales

    • name: Identificador único del paquete (vendor/package)
    • require: Dependencias necesarias para producción
    • require-dev: Dependencias solo para desarrollo (tests, linters, etc.)
    • autoload: Configuración del cargador automático de clases
    • license: Licencia del proyecto (MIT, Apache, etc.)

    3.4. Versiones en composer.json

    Ejemplos de especificación de versiones:

    
{
    "require": {
        "monolog/monolog": "1.0.0",           // Versión exacta
        "symfony/console": "^5.0",             // Máximo ^5.x (compatible)
        "phpunit/phpunit": "~9.5",             // Máximo ~9.x
        "laravel/framework": ">=8.0",          // Versión 8.0 o superior
        "doctrine/orm": "2.7.*",               // Versión 2.7.x
        "guzzlehttp/guzzle": "^7.0 || ^8.0"  // Versión 7.x o 8.x
    }
}

    4. Instalar y gestionar dependencias

    4.1. Instalar dependencias

    
# Instala todas las dependencias definidas en composer.json
composer install

# Crea dos archivos importantes:
# - composer.lock: Versiones exactas instaladas (para reproducibilidad)
# - vendor/: Carpeta con todos los paquetes descargados

    4.2. Añadir nuevas dependencias

    
# Instala una nueva dependencia y la añade a composer.json
composer require monolog/monolog

# Con versión específica
composer require symfony/console:^6.0

# Múltiples paquetes
composer require phpunit/phpunit guzzlehttp/guzzle

    4.3. Dependencias de desarrollo

    
# Instala como dependencia de desarrollo (--dev o -d)
composer require --dev phpunit/phpunit

# Estas dependencias no se instalan en producción

    4.4. Actualizar dependencias

    
# Actualiza todas las dependencias
composer update

# Actualiza un paquete específico
composer update symfony/console

# Actualiza y bloquea versiones en composer.lock

    4.5. Eliminar dependencias

    
# Elimina un paquete
composer remove monolog/monolog

# Elimina dependencia de desarrollo
composer remove --dev phpunit/phpunit

    4.6. Verificar dependencias

    
# Muestra todas las dependencias instaladas
composer show

# Muestra información de un paquete específico
composer show monolog/monolog

# Valida composer.json y composer.lock
composer validate

    5. Autoload y PSR-4

    5.1. ¿Qué es el Autoload?

    El autoload es un mecanismo que carga automáticamente las clases PHP cuando las necesitas, sin tener que usar require() o include() manualmente.

    5.2. Sin Composer (forma antigua)

    
<?php
// Sin autoload, hay que incluir manualmente cada archivo
require 'src/Usuario.php';
require 'src/Producto.php';
require 'src/Database/Conexion.php';
require 'vendor/monolog/monolog/src/Monolog/Logger.php';
require 'vendor/monolog/monolog/src/Monolog/Handler/StreamHandler.php';
// ... y muchos más archivos

// Luego usar las clases
$usuario = new Usuario();
$producto = new Producto();
$logger = new Logger('app');
?>

    5.3. Con Composer y Autoload (forma moderna)

    
<?php
// Una sola línea de autoload de Composer
require 'vendor/autoload.php';

// Y ya puedes usar todas las clases automáticamente
$usuario = new App\Usuario();
$producto = new App\Producto();
$conexion = new App\Database\Conexion();
$logger = new Monolog\Logger('app');
?>

    5.4. ¿Qué es PSR-4?

    PSR-4 (PHP Standards Recommendation 4) es un estándar que define cómo organizar y nombrar el espacio de nombres (namespace) de tus clases para que Composer las encuentre automáticamente.

    PSR-4 mapea namespaces a directorios del proyecto:

    • Namespace: App\Usuario → Archivo: src/Usuario.php
    • Namespace: App\Database\Conexion → Archivo: src/Database/Conexion.php
    • Namespace: App\Models\User → Archivo: src/Models/User.php

    6. Configurar PSR-4 en composer.json

    6.1. Estructura básica

    
{
    "name": "mi-app/proyecto",
    "require": {
        "php": ">=7.4"
    },
    "autoload": {
        "psr-4": {
            "App\\": "src/",
            "App\\Models\\": "src/Models/",
            "Tests\\": "tests/"
        }
    }
}

    6.2. Explicación de PSR-4

    La sintaxis "Namespace\\": "directorio/" significa:

    • "App\\": El namespace raíz es App
    • "src/": Mapea al directorio src
    • Una clase en src/Usuario.php tendrá namespace App\Usuario
    • Una clase en src/Database/Conexion.php tendrá namespace App\Database\Conexion

    6.3. Estructura de proyecto recomendada

    
mi-proyecto/
├── src/                      # Código de la aplicación
│   ├── Usuario.php
│   ├── Producto.php
│   ├── Database/
│   │   └── Conexion.php
│   └── Models/
│       ├── User.php
│       └── Post.php
├── tests/                    # Tests unitarios
│   ├── UserTest.php
│   └── ProductoTest.php
├── vendor/                   # Dependencias (generado por Composer)
├── composer.json
├── composer.lock
└── index.php

    6.4. Ejemplo de archivo Usuario.php

    
<?php
// src/Usuario.php
namespace App;

class Usuario {
    private $nombre;
    private $email;
    
    public function __construct($nombre, $email) {
        $this->nombre = $nombre;
        $this->email = $email;
    }
    
    public function getNombre() {
        return $this->nombre;
    }
    
    public function getEmail() {
        return $this->email;
    }
}
?>

    6.5. Usar la clase desde index.php

    
<?php
// index.php
require 'vendor/autoload.php';

// Usar la clase Usuario - se carga automáticamente gracias a PSR-4
$usuario = new App\Usuario('Juan', 'juan@ejemplo.com');

echo "Nombre: " . $usuario->getNombre() . "<br>";
echo "Email: " . $usuario->getEmail() . "<br>";
?>

    7. Casos de uso avanzados

    7.1. Múltiples namespaces

    
{
    "autoload": {
        "psr-4": {
            "App\\": "app/",
            "Src\\": "src/",
            "Config\\": "config/",
            "Tests\\": "tests/"
        }
    }
}

    7.2. Autoload de archivos simples

    
{
    "autoload": {
        "files": [
            "helpers/funciones.php",
            "helpers/constantes.php"
        ],
        "psr-4": {
            "App\\": "src/"
        }
    }
}

    7.3. Autoload de desarrollo

    
{
    "autoload": {
        "psr-4": {
            "App\\": "src/"
        }
    },
    "autoload-dev": {
        "psr-4": {
            "Tests\\": "tests/"
        }
    }
}

    7.4. PSR-0 (legacy)

    PSR-0 es el estándar anterior a PSR-4. Hoy en día se recomienda usar PSR-4. 

    
{
    "autoload": {
        "psr-0": {
            "Vendor_Prefix_": "src/"
        }
    }
}

    8. Regenerar el autoload

    8.1. Cuándo regenerar

    Después de cambiar la configuración de autoload en composer.json, debes regenerar el archivo de autoload: 

    
# Regenera el archivo vendor/autoload.php
composer dump-autoload

# Opción: Optimizar para producción
composer dump-autoload --optimize
composer dump-autoload -o

    8.2. Optimizar para producción

    
# Usa classmaps en producción (más rápido)
composer install --no-dev --optimize-autoloader

# Usa el archivo optimizado
composer dump-autoload --optimize --no-dev

    9. Entender vendor/autoload.php

    9.1. ¿Qué es vendor/autoload.php?

    vendor/autoload.php es un archivo generado automáticamente por Composer que:

    • Define una función de carga automática (autoloader) de Composer
    • Carga el autoloader de cada dependencia
    • Registra los namespaces según tu configuración PSR-4

    9.2. Estructura de vendor/

    
vendor/
├── autoload.php                  # SIEMPRE incluye este archivo
├── composer/
│   ├── autoload_classmap.php     # Mapa de clases (producción)
│   ├── autoload_files.php        # Archivos automáticos
│   ├── autoload_namespaces.php   # Namespaces PSR-0
│   ├── autoload_psr4.php         # Namespaces PSR-4
│   └── installed.json            # Información de paquetes
├── monolog/                      # Paquete 1
├── symfony/                      # Paquete 2
└── ...

    9.3. El archivo composer/autoload_psr4.php (generado)

    
<?php

// autoload_psr4.php @generated by Composer

$vendorDir = dirname(__DIR__);
$baseDir = dirname($vendorDir);

return array(
    'Tests\\' => array($baseDir . '/tests'),
    'App\\' => array($baseDir . '/src'),
    'Monolog\\' => array($vendorDir . '/monolog/monolog/src'),
    'Symfony\\' => array($vendorDir . '/symfony/console/src'),
    // ... más namespaces
);
?>

    9.4. No modificar archivos generados

    NUNCA edites los archivos en vendor/ o vendor/composer/ manualmente. Se regeneran automáticamente y perderías los cambios.

    Si necesitas cambiar algo, modifica composer.json y ejecuta composer update o composer dump-autoload.

    10. Casos comunes y soluciones

    10.1. "Class not found" - La clase no se encuentra

    Problema: Composer no encuentra una clase que debería existir.

    Solución:

    1. Verifica que el archivo existe en la ubicación correcta
    2. Verifica que el namespace coincide con la configuración PSR-4
    3. Regenera el autoload: composer dump-autoload
    4. Limpia la caché: composer clear-cache
    
# Ejemplo: si tienes src/Usuario.php con namespace App\
# Asegúrate de que en composer.json está: "App\\": "src/"
# Y que el archivo empieza con: <?php namespace App;

    10.2. Conflictos de dependencias

    Problema: Dos paquetes requieren versiones incompatibles del mismo paquete.

    Solución:

    • Revisa el error que muestra Composer
    • Considera usar una versión compatible
    • Actualiza los paquetes a versiones más nuevas

    10.3. composer.lock no sincronizado

    Problema: El equipo de desarrollo tiene versiones diferentes de paquetes.

    Solución:

    
# Elimina vendor/ y composer.lock
rm -rf vendor/ composer.lock

# Reinstala exactamente como está definido
composer install

    10.4. Memoria insuficiente

    Problema: "Fatal error: Allowed memory size exhausted"

    Solución:

    
# Aumenta el límite de memoria
php -d memory_limit=-1 composer install

# O establécelo en php.ini
memory_limit = 2G

    10.5. No cargar dependencias de desarrollo en producción

    
# En desarrollo
composer install

# En producción
composer install --no-dev

    11. Mejores prácticas con Composer

    11.1. Control de versiones (git)

    • Incluir: composer.json y composer.lock
    • Excluir: vendor/ directorio (muy grande)
    
# En .gitignore
/vendor/

    11.2. composer.lock en repositorio

    Siempre incluye composer.lock en tu repositorio Git. Esto garantiza que todos los desarrolladores usen exactamente las mismas versiones de dependencias.

    11.3. Actualizar versus instalar

    • composer install: Instala exactamente lo que está en composer.lock
    • composer update: Actualiza dependencias (crea nuevo composer.lock)

    11.4. Especificar versiones

    
{
    "require": {
        "monolog/monolog": "^2.0",          // Recomendado para librerías
        "laravel/framework": ">=8.0,<9.0",  // Alternativa segura
        "phpunit/phpunit": "~9.5"           // ~9.5 = >=9.5,<10.0
    }
}

    11.5. Organización de código

    • Usa namespaces consistentes con la estructura de directorios
    • Un namespace por archivo (generalmente)
    • Nombres de clases deben ser CamelCase
    • Nombres de archivos deben coincidir con el nombre de la clase

    11.6. Documentar dependencias

    
{
    "name": "miempresa/proyecto",
    "description": "Descripción clara del proyecto",
    "require": {
        "php": ">=7.4",
        "monolog/monolog": "^2.0"  // Logging
    },
    "require-dev": {
        "phpunit/phpunit": "^9.0"   // Testing
    }
}

composer require paquete       # Añadir dependencia

# Desarrollo
composer require --dev phpunit/phpunit  # Para testing
composer require --dev squizlabs/php_codesniffer  # Para análisis

    13. Resumen

    13.1. Puntos clave

    • Composer es el gestor de dependencias estándar en PHP
    • composer.json define las dependencias de tu proyecto
    • vendor/autoload.php carga todas las clases automáticamente
    • PSR-4 es el estándar para organizar namespaces y clases
    • Autoload elimina la necesidad de require/include manuales

    13.2. Flujo típico de trabajo

    1. Ejecuta composer init para crear composer.json
    2. Configura autoload con PSR-4 en tu composer.json
    3. Ejecuta composer install para instalar dependencias
    4. Incluye require 'vendor/autoload.php' en tu código
    5. Usa las clases con sus namespaces automáticamente

    13.3. Estructura recomendada

    
proyecto/
├── src/                       # Tu código (PSR-4)
│   ├── Models/
│   └── Controllers/
├── tests/                     # Tests
├── vendor/                    # Dependencias (generado)
├── .gitignore
├── composer.json
└── composer.lock

    13.4. Comandos esenciales

    • composer init - Crear nuevo proyecto
    • composer install - Instalar dependencias
    • composer require paquete - Añadir dependencia
    • composer update - Actualizar todo
    • composer dump-autoload - Regenerar autoload
